# Evolve mode

Use this mode when a stable owner question or named consumer has a confirmed
no-fit or failure in the admitted stats surface. Missing gates block evolve;
they do not turn it into answer or diagnosis.

## Inputs

- stable owner question and owner
- named consumer
- observed no-fit or failure with source refs
- requested authority ceiling and explicit effect authority
- current profile, operation, port, mechanic, or candidate route

## Procedure

1. Stop `blocked_missing_meaning` when owner question, consumer, no-fit, or
   authority ceiling is absent.
2. Use `source-return.md` to resolve one source and reciprocal mechanic route.
3. Decide the owner before the shape:
   - domain meaning and freshness expectation stay in the domain owner's port
   - shared compatibility or receipt vocabulary belongs to the matching
     `stats/` family
   - a derived public question needs an authored profile or operation
   - repeatable projection or refresh belongs to the reciprocal mechanic
   - pure implementation stays separate from I/O and transport
4. Compare reuse, extension, activation, retirement, and no-change. Do not add
   a central surface when an owner-local answer or existing profile suffices.
5. Exercise expected, rejected, missing, stale, incompatible, and nearest
   collision cases manually before durable automation.
6. Add a test or validator only when those trials establish a stable long-lived
   invariant and the active task authorizes it.
7. With effect authority, change authored source before generated companions,
   rebuild only declared projections, inspect artifacts manually, and report
   what green checks cannot prove. Without authority, return a bounded
   owner-ready proposal.
8. Return a `bounded-stats-evolution-result` with decision, owner, sources,
   affected consumers, cases, effect posture, artifacts, verification, claim
   limit, handoff, and stop line.

## Undefined cross-owner score

When a consumer requests one numeric score across several owner publishers but
the owners have not authored freshness windows, source identity, and an
aggregation rule:

- select `evolve` because the operation requests a new owner surface
- return `decision: no_change` with
  `blocker: blocked_missing_meaning`
- keep the named consumer and each publisher owner visible
- hand freshness expectation to the publisher owners and score meaning to the
  consumer owner
- do not create a central profile, sidecar, registry field, schema, score
  range, threshold, weighting, default, or missing-value substitution
- distinguish per-publisher categorical cases (`within_window`, `stale`,
  `missing`, `invalid`, `window_unknown`) without assigning numeric outcomes
- require a later cross-owner compatibility review only after owner contracts
  exist; that review may still decide that no global aggregation is meaningful

Resolve this structural no-fit from the exact source-coverage profile, live
source registry, live-refresh contract, and source-coverage mechanic contract.
The exact fourth path is
`mechanics/audit/parts/source-coverage/CONTRACT.md`. Stop owner reads after
those four authored sources.
Do not probe `state/`, a deployed runtime, committed/generated outputs, the
schema, validator, source-home manifest, or full mechanic topology: none can
supply the missing owner meaning. Do not run `rg --files` or read the
source-coverage part's tests, `VALIDATION.md`, or docs.

The result must explicitly report `effects: none`, `artifacts: none`, the
owner sources inspected, the cases considered, the absence of accepted
aggregation meaning, the owner handoffs, and the stop line. Use these exact
top-level result fields:

```text
decision
blocker
owner
affected_consumers
sources
cases
effects
artifacts
verification
claim_limit
handoff
stop_line
```

A list of future cases without those ABI fields is incomplete.

## Termination

Stop after the smallest owner-first change, proposed change, `no_change`, or
explicit blocker. Never widen into a dashboard, global score, workflow,
runtime repair, or central owner-metric invention.
